> **v3 迭代（第二轮反馈：还是太简单）**：目标扩到 **≥50 关 + 更多创造性机制（每个新机制仅一关引入）**，并保持不规则、减提示、各阶段提交。新增 3 种确定性机制：**单向格（箭头，只能顺向进入）、脆地（玩家离开即塌成深坑，仅一次通过）、钥匙/锁（拾钥开同色锁）**。为高效产出大量「可解的难关」，写了**逆向拉箱生成器**（记录解法，故即便超出求解器上限也能测）。进度见 §6。

> **v2 迭代（试玩反馈后）**：试玩反馈「太简单、引入太啰嗦、地图太方正」。据此：① 新增第六种机制**折跃门**（仅传送玩家、箱子不可过——制造「够不到推箱方位」的探索）；② 关卡从 6 扩到 **20**（六章），全部不规则地形 + 立柱/深坑障碍，难度递增；③ 大幅删减提示——仅每种机制**首次出现**给一行规则（不给解法），其余关卡零提示，另设「玩法/图例」面板随时可查；④ 菜单按章节分组。详见 §5。

## 1. 玩法设计（核心创意）

经典推箱子的「基调」保留：网格、不可拉只能推、把箱子推到目标点。**拓展才是重点**——每一关引入一种新机制，最终关合流。机制之间彼此正交、可组合，制造远超经典推箱子的策略空间。

### 1.1 五种地形 / 机制

| 机制 | 规则 | 带来的策略维度 |
|---|---|---|
| **目标点 Goal** | 把箱子推到目标点，全部覆盖即过关 | 经典基线 |
| **冰面 Ice `~`** | **箱子很沉**：被推上冰面后会沿推动方向一直滑行，直到撞到障碍或滑出冰面才停下。**玩家穿防滑靴，不受冰影响**（可在冰上正常行走） | 一次推动就「锁定」了一条轨迹，必须预判落点；墙/目标点可当「刹车」 |
| **深坑 Pit `^`** | 玩家不能踏入深坑（视作墙）；把箱子推入坑中会**填平**该坑（箱子消耗、坑变成可通行的地面） | 箱子是稀缺资源：哪些去填坑搭路、哪些去目标点，是取舍博弈 |
| **压力板 + 闸门 Plate/Gate** | 闸门默认关闭（挡路）。当其关联的压力板被**重物（玩家或箱子）**压住时闸门开启。松开即关闭 | 玩家只有一个身位 → 必须「停一个箱子在板上」长期顶门，再用别的箱子穿门，制造顺序与资源占用 |
| **颜色匹配 Colored** | 彩色箱子必须推到**同色**目标点；任意色目标点接受任意箱子 | 指派问题：在狭窄通道里，哪个箱子去哪个目标点不能乱来 |

> 关键设计选择：**只有箱子在冰上滑，玩家不滑。** 这样规则可教学性强（一句话："箱子沉、你有防滑靴"），冰的策略乐趣（预判轨迹、不可逆的一推）完整保留，又避免了玩家滑行带来的导航挫败与压力板/闸门交互的歧义。

### 1.2 关卡曲线（≥6 关，逐关引入 + 终章合流）

| 关 | 标题 | 引入机制 | 设计意图 |
|---|---|---|---|
| L1 | 起步 First Steps | 纯经典 | 教学：移动、推、撤销、重开 |
| L2 | 薄冰 Thin Ice | 冰面 | 用墙/目标点当刹车，预判滑行 |
| L3 | 深坑 The Gap | 深坑 | 箱子取舍：填坑 vs 达标 |
| L4 | 机关 Mechanism | 压力板 + 闸门 | 用箱子顶门、规划顺序 |
| L5 | 色彩 Hues | 颜色匹配 | 指派 + 路径，避免互相挡道 |
| L6 | 合流 Confluence | 冰 + 坑 + 闸门 + 颜色 | 终章：综合调度 |

### 1.3 反馈与「内涵」

- **撤销 / 重开**：纯逻辑谜题不应有"死局只能重来"的惩罚感。提供无限撤销与一键重开。
- **步数 / 推动数**：记录步数与推动次数，给出每关「最优解参考（par）」（由内置求解器算出）作为可选挑战，而非强制压力。
- **无死亡状态**：玩家不会踩坑致死（坑对玩家=墙），避免突兀重置。失败仅来自「把局面推成无解死局」，可随时撤销。

## 2. 技术栈与架构

> 原则：**少依赖、可静态化、单进程可跑、易部署**（2 核 8G 绰绰有余）。工程上体现"高级"——共享规则引擎、类型安全、服务端可复算校验。

### 2.1 总体形态

单包（single-package）TypeScript 工程，三个逻辑层共享同一套**纯函数规则引擎**：

```
src/engine/   纯 TS 规则引擎（无框架依赖）——前后端共享
  types.ts    数据模型
  level.ts    关卡解析（ASCII + 机关元数据 → 运行态）
  rules.ts    一步移动 / 推动 / 冰滑 / 填坑 / 闸门 / 胜负判定
  levels.ts   6 关数据
  solver.ts   BFS 求解器（验证可解性 + 算 par + 死锁检测）
src/web/      前端（Vite + TS，无 UI 框架，DOM/CSS 渲染）
server/       Fastify 后端（托管静态 + API + 服务端校验 + SQLite 持久化）
scripts/      verify-levels.ts —— 对全部关卡跑求解器的自测脚本
```

### 2.2 前端

- **构建**：Vite + TypeScript，**不引入 React/Vue** —— 谜题渲染是一个小网格，框架是负担。手写轻量渲染层即可，产物更小、首屏更快、也更"克制"。
- **渲染**：**DOM + CSS Grid**，棋子用绝对定位 + `transform` 平滑过渡（`cubic-bezier` 缓动）。相比 Canvas：可访问性好、动画用 CSS 即可、调试简单、像素锐利。
- **美术方向（拒绝 AI 风味）**：遵循极简游戏美学（Monument Valley / Mini Metro：基本几何形、<15 色、留白、外科手术式用色）。
  - **不下载卡通 sprite**，全部用 CSS/SVG 绘制几何形——这恰恰是"高级感"的来源，也保证离线部署、零版权风险。
  - 配色：暖色"纸与墨"基调（米白 `#f4f1ea` 底、暖炭墙、木色箱），机制色克制（冰=冷雾蓝、坑=内凹深色、压力板=内嵌环、彩色箱用低饱和大地色：陶土/鼠尾草/灰蓝/赭石）。
  - 字体：高质量 system serif/grotesque 栈，不依赖 CDN，保证离线与一致性。
  - 交互：键盘（方向键/WASD）、点击/触摸（点目标格自动寻路一步式或方向滑动）、撤销/重开按钮，过渡有"重量感"的缓动。
- **状态**：本地进度（已通关、最佳步数）存 `localStorage`；可选上传到后端排行榜。

### 2.3 后端

- **运行时**：Node.js + **Fastify**（比 Express 更快更现代，依赖少）。单进程同时：
  1. 托管前端构建产物（`dist/`）；
  2. 提供 API：关卡目录、提交解法、最佳成绩排行。
- **服务端权威校验**：后端**复用同一套 `engine`**，对客户端提交的移动序列重放，确认确实通关并算出步数——杜绝伪造成绩，保证"策略完整实现"在服务端也成立。
- **持久化**：**SQLite（better-sqlite3）**——文件型、零外部服务、单机极快，最适合 2 核 8G 小机。存每关最佳成绩 / 排行。
- **构建/部署**：`vite build` → `dist/`（静态）；`esbuild` 把 `server` + `engine` 打成单文件 `dist-server/index.js`；生产仅 `node dist-server/index.js` 一个进程。配 systemd/pm2 + 可选 nginx 反代。

### 2.4 为什么这套"够高级又能过"

- 共享规则引擎 = 客户端体验与服务端校验**同一份真理来源**，逻辑零漂移。
- 内置 BFS 求解器既是**关卡 QA**（保证每关可解、给出 par、检测死锁），也是自测基础设施——直接回应"每关自测、查缺补漏"。
- 全静态前端 + 单 Node 进程 + SQLite = 部署面极小，2 核 8G 轻松。

### 阶段 4–5 决策记录

- **前端零框架**：网格用 CSS Grid，玩家/箱子用绝对定位 + `transform` 过渡——走一步是短过渡，冰面滑行是「按距离拉长的缓出过渡」，沉坑是「滑到坑位 → 缩放淡出 → 移除元素 + 坑标记为已填」。产物仅 ~15.6KB JS（gzip 6.5KB）/ 9.8KB CSS，且**不打包求解器**。
- **服务端权威校验**：`POST /api/scores` 不信任客户端上报的步数，而是用**同一套引擎 `replay()`** 重放提交的移动序列，只有真正通关才入库，步数/推动数以重放结果为准 → 杜绝刷分。
- **修了一个部署暗坑**：原用 `HOST` 环境变量做绑定地址，但很多 shell 会导出 `HOST=主机名`，导致服务端绑定错地址。改名为 `BIND_HOST`，默认 `0.0.0.0`。

### 阶段 2–3 决策记录

- **持久化改用 Node 24 内置 `node:sqlite`**：真 SQLite、零原生编译（免去 `better-sqlite3` 在 Windows 上的构建风险），仅一句实验性告警。
- **求解器从 BFS 升级为 A\***：纯 BFS 在 3 箱以上关卡（L5/L6）状态爆炸打到上限。A\* 用「目标点↔同色箱子的最小指派曼哈顿和」作启发（同时充当死局剪枝：所需箱子全部沉坑 → +∞）。
  - 该启发在**无冰**关上可采纳 → 步数最优；在**冰面**关（L2）会高估（一次推动可滑多格）→ 返回的是「可达的强解」而非理论最短。L2 的真实最优由早期 BFS 证得为 14，已写入 `par`。
- **关卡两处设计 bug 由求解器抓出并修复**：L5 角落死箱（箱子贴墙无法推动）、L6 过于开阔导致搜索爆炸 → 收紧为走廊式终章。

| L1 起步 | ✓ | 10 | 5 | 121 |

### 4.2 `npm run smoke:api` — 后端逻辑（fastify.inject，无需端口）
✓ health；✓ levels=6；✓ 六关「真实解」均被接受且服务端重算步数与求解器一致（10/17/32/11/41/26）；✓ 伪造解 400；✓ 未知关 400；✓ 非法方向 400；✓ 排行榜正确回显。

### 4.3 `npm run smoke:ui` — 真实 UI（jsdom，模拟按键打通每关）
逐关从菜单进入 → 用求解器解法模拟方向键 → 过关浮层出现。六关全过，箱子数 2/2/3/2/3/4 正确，步数计数正确。捕获/验证了输入处理、渲染对账、过关流程、关卡导航。

### 4.4 真实本地服务器 HTTP 自测（端口 8799，已关闭）
`node dist-server/index.js` 实跑：✓ `GET /api/health`→`{ok,levels:6}`；✓ `GET /api/levels`（含 par）；✓ `GET /` 返回 index.html；✓ 哈希资产 200；✓ 伪造 `POST /api/scores`→400；✓ SPA 回退→200。
**端口关闭确认**：自测后用 `Get-NetTCPConnection -LocalPort 8799` 定位 OwningProcess 并 `Stop-Process`，复查 `PORT 8799 CLOSED OK`。（首次用 shell `$!` 误杀了 MSYS 壳 PID 而非 node 真 PID，已改为按端口杀进程。）

### 4.5 真实浏览器视觉自测（Playwright 截图，端口 8801，已关闭）
用 Playwright（Chromium headless）实跑构建产物并截图菜单 + 全部六关，逐张人眼审查。
- **抓到并修复一个真实视觉 bug**：箱子渲染成「小点」而非木箱。根因——`.crate` 用了 `padding: 11%`，但 `.piece` 是绝对定位元素，**百分比 padding 是相对「包含块（整块棋盘）」而非自身**解析的，于是 padding≈一整格把 body 压没了。玩家方块没事是因为它用绝对定位的 `inset`（相对 piece 解析）。改为给 `.crate .body` 用 `position:absolute; inset:11%`。
- 复测：菜单、L1–L6 均渲染正确——木箱有倒角与重量感、压力板/闸门同组配色、深坑内凹可辨、彩色箱与同色目标环清晰、最宽的 L3（12 格）自适应不溢出。
- 截图脚本通过 `child_process.spawn` 持有 node 真句柄，结束 `child.kill()`；复查 `PORT 8801 CLOSED OK`。Playwright 以 `--no-save` 安装，不进 `package.json`，保持部署依赖精简。

> 已知良性现象：`node:sqlite` 启动有一行 ExperimentalWarning；不影响功能。

## 5. v2 迭代：折跃门 + 20 关 + 减提示（试玩反馈后）

### 5.1 新机制：折跃门（player-only portal）
- 设计取舍：箱子穿门会引发「推箱过门」的歧义（玩家进门口会被传走、与刚传出的箱子撞位）。为保持**确定性**，定为**只传送玩家、箱子把折跃门当墙**。这反而是更纯的探索机制——「我够不到这个箱子的可推一侧，得先借门绕过去」。
- 引擎改动：`Cell.portal` + `Level.portalPartner[]`；`crateCanEnter` 对折跃门返回 false；玩家踏入折跃门→落到同色伙伴格（伙伴格须可入）。渲染用同心环 + 玩家瞬移的 `warp` 动画（不跨屏滑行）。求解器经 `applyMove` 自动兼容。

### 5.2 关卡：6 → 20（六章），靠求解器迭代
- 用一个临时单图测试器（`tsx` + 求解器，<1s/图）反复试错，再把通过的图并入目录。**首轮 20 张草稿 14 张不可解**——常见坑：贴墙角的死箱、顶行的箱子无法下推、彩箱在角、目标数≠箱数、单宽走廊里箱子堵死折跃门路。逐一修复至全绿。

### 6.1 三种新机制（确定性，引擎/渲染/图例/求解器均已接入）

### 6.2 逆向拉箱生成器 `scripts/gen.ts`（关键产能工具）

- **踩过两个大坑**：
  1. **拖尾参差行**导致边界漏成地板 → `wall()` 把未指定格当墙。

- 产出 42 张古典硬关（par 18–40，5–6 箱，不规则），全部求解器最优且自带已验证解法。

### 6.3 目录：23 设计关 + 28 生成关 = **51 关 / 10 章**

## 9. v6 迭代：2.5D 立体谜题（进行中，分阶段）

### 9.1 v6 设计判断（方案选型）
**1）真 3D / 2.5D 等轴 / 2D 引擎+3D 表现——选哪种？**
选 **「2D 逻辑引擎 + 整数高度维度（height/z）+ 真 2.5D 等轴渲染（CSS 3D transforms）」**。理由：本作的命门是「纯函数引擎 + 服务端 token 重放权威校验 + A* 求解器 + jsdom 真 UI 自测」四件套；把逻辑保持为**带高度的网格**（确定性、可重放、可求解、可测），只在**表现层**做真三维投影，能在不破坏任何一件套的前提下拿到真实的立体空间玩法（高低差、台阶、斜坡、桥、升降、垂直折跃）。自由旋转的真 3D 既会带来「迷路」（题面已警示），又会撑大包体、还让 jsdom 测不了。

**2）为何不用 Three.js？如何保证不是「伪 3D 阴影」？**
不引入 Three.js：① 包体（gz ~150KB+）违背「控制包体、少依赖、轻部署」与本项目一贯精简；② WebGL 在 jsdom 下无法渲染 → 会直接打断 `smoke:ui`（驱动真实 UI 的测试）；③ 谜题只需固定等轴相机，自由相机反而坏事。**不伪 3D 的做法**：棋盘用固定 `rotateX/rotateZ` 的等轴投影 + `transform-style: preserve-3d`，每个格子/棋子按 `height` 做真实 `translateZ` 并绘出**顶面 + 侧面**（被挤出的几何体，有真实的高度、堆叠与遮挡），而非平面网格 + box-shadow。遮挡用 z 排序 + 关键物体描边/半透明/抬高处理。

**3）如何不破坏服务端重放、求解器、自测？**
引擎仍是 `GameState` 上的纯函数。`Cell` 加 `height` 与斜坡；`applyMove` 变为**高度感知**（低处不能直接推上高台、推下台阶会下落到低处、玩家上坡需斜坡、可自由下台阶）。`stateKey` 在需要时纳入高度。`replay/replayDiptych` 形态不变（仍是 token 序列），服务端权威校验照旧。**52 关全部 height=0 无斜坡 → 行为与现状完全一致**，故旧关与三层自测全绿不变；新机制全部补 replay 校验。

**4）哪些旧机制迁移到 3D / 哪些留经典？**
迁移：推/拉、坑（跨高度的下落填平）、折跃（升级为**垂直/楼层折跃**）、重力倾斜（叠加高度更有戏）、桥（新）、**两边同时推/双端受力**（v6 头牌，双联画协作精神的 3D 进化）。留作经典平面：冰面、镜面、双联画原版——旧 52 关整体保留为 **Classic/平面模式**（`mode:'2d'`），新 3D 关进新章节（按是否含高度数据自动选 2D 网格 / 等轴渲染）。

**5）关卡数量依据：**
新增 3D 章节按**质量优先**控制在 **~20–24 关**（题面建议 18–30）：高度/斜坡引入（3–4）、桥与断桥（3）、升降/重量（3）、**双端受力章 ≥5**（递增 + 含 1 关高度组合 + 1 关双联组合 + 1 关 boss）、垂直折跃（2–3）、组合关（3–4）。每机制 1–2 关引入，深度靠组合而非堆量；旧 52 关不动。

### 9.3 等轴 2.5D 渲染层

**Renderer 设计 / 为什么没用 Three.js / CSS 实现方式**
- 新增**独立** `src/web/iso.ts` 的 `IsoRenderer`，**不改动** 2D `BoardRenderer`。`level.is3D`（parseLevel 自动判定：任一格 `height>0`）决定 `playLevel` 选哪个渲染器；旧 52 平面关继续走 2D 网格，**零改动**。
- 不用 Three.js：WebGL 在 jsdom 下无法渲染会打断 `smoke:ui`；包体 +150KB gz 违背精简；固定相机谜题无需自由 3D。当前产物 JS 61KB/gz 17.5KB，仍无 Three.js。
- **CSS 实现**：2:1 等轴**屏幕投影**（`sx=(x−y)·TW/2`，`sy=(x+y)·TH/2 − layer·ZH`），每个 tile = 一个菱形 **top face**（clip-path）+ 两个 `skewY(±26.565°)` 的平行四边形**侧面**（真实挤出几何，**非 box-shadow**）。按格深度 `(x+y)·1000+layer·10` 做**画家算法 z-index**，得到可靠遮挡（CSS 3D 无逐像素 z-buffer，故不用 rotateX 整盘旋转那一套）。棋子（小人 SVG / 木箱小立方）也按投影定位，**高度变化平滑过渡**（动画 left/top，不瞬移）。
- **遮挡**：墙体高出地面 1.7 层，互相遮挡正确；当玩家被前方更高的格挡住时，叠加**半透明透视的玩家 + 虚线轮廓 ghost**（`updateGhost` 启发式：检测前方/对角格是否高出玩家 ≥1.4 层）。

**美术**：小人在等轴下为有体积的小棋子（大头/亮眼/腮红/天线，带朝向：背身隐眼、推/走有微动）；木箱为顶面+两侧面的小立方；地板/墙/坑/目标环/折跃门同心环/脆地/镜面在等轴顶面均可辨认。

**stepped visual test level（dev-only，不计入正式关数）**：`src/web/demo.ts` 的 `立体演示`，菜单底部「▦ 立体演示 (dev)」隐藏入口进入。含：0/1/2 台阶、折跃门把玩家**升到高台**、把箱子**沿 2→1→0 台阶推下**到目标、一段高墙遮挡、对照旁边的 Classic 平面关（l1）。求解器证明可解 par 5（也用于过关浮层不被 3D 棋盘遮挡的验收）。

**测试与验收（区分两类）**
- 逻辑/流程（自动）：`typecheck`、`verify 52/52`、`smoke:api`、`smoke:ui` 全绿——证明 DOM/输入/状态/过关流程与 52 平面关不破。**`smoke:ui` 不证明 CSS 3D 视觉**。

### 9.4 3D 机制内核 + 第一批正式 3D 关卡（Stage 4）

**新增机制规则（确定性，已接入 verify/replay/服务端校验；52 平面关字节级不变）**
- **Ramp 斜坡**（地图字符 `n/e/s/w` = 上坡朝向）：一格斜坡把它的基准高度 `h`（−坡向边）连到 `h+1`（+坡向边），**只能沿坡轴进出**（垂直方向不能上下坡）。玩家与箱子都能顺坡上一层。规则由 `edgeHeight(cell, edgeDir)` 给出每条边的高度，`canClimb` 要求「出边高 ≥ 入边高」（可平/可下、不可上，斜坡除外）。
- **Bridge 索桥**（字符 `=`）：可通行地形，但**只承得住玩家**，箱子推不上去（`crateCanEnter` 对 bridge 返回 false）。用于「过桥绕到箱子另一侧去推」。
- **Lift 升降台**（字符 `T`）：占用（玩家或箱子站上）即升 `LIFT_RISE=2` 层，离开即落。高度**由占用情况推导**（非锁存），因此位置已在 `stateKey` 中 → 求解/去重/replay 天然正确，无需额外状态字段。等轴渲染按占用动态升降并平滑过渡。
- **Vertical portal 竖直折跃**：沿用既有折跃门（连接任意高度、只送玩家、出口被占则不传送），无需改引擎；3d4 正式化为「升到高台另一侧」。
