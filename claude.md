## 1. 玩法设计（核心创意）

经典推箱子的「基调」保留：网格、不可拉只能推、把箱子推到目标点。**拓展才是重点**——每一关引入一种新机制，最终关合流。机制之间彼此正交、可组合，制造远超经典推箱子的策略空间。

### 1.1 五种地形 / 机制

| 机制 | 规则 | 带来的策略维度 |
|---|---|---|
| **目标点 Goal** | 把箱子推到目标点，全部覆盖即过关 | 经典基线 |
| **冰面 Ice `~`** | **箱子很沉**：被推上冰面后会沿推动方向一直滑行，直到撞到障碍或滑出冰面才停下。**玩家穿防滑靴，不受冰影响**（可在冰上正常行走） | 一次推动就「锁定」了一条轨迹，必须预判落点；墙/目标点可当「刹车」 |
| **深坑 Pit `^`** | 玩家不能踏入深坑（视作墙）；把箱子推入坑中会**填平**该坑（箱子消耗、坑变成可通行的地面） | 箱子是稀缺资源：哪些去填坑搭路、哪些去目标点，是取舍博弈 |
| **压力板 + 闸门 Plate/Gate** | 闸门默认关闭（挡路）。当其关联的压力板被**重物（玩家或箱子）**压住时闸门开启。松开即关闭 | 玩家只有一个身位 → 必须「停一个箱子在板上」长期顶门，再用别的箱子穿门，制造顺序与资源占用 |
| **颜色匹配 Colored** | 彩色箱子必须推到**同色**目标点；任意色目标点接受任意箱子 | 指派问题：在狭窄通道里，哪个箱子去哪个目标点不能乱来 |

> 关键设计选择：**只有箱子在冰上滑，玩家不滑。** 这样规则可教学性强（一句话："箱子沉、你有防滑靴"），冰的策略乐趣（预判轨迹、不可逆的一推）完整保留，又避免了玩家滑行带来的导航挫败与压力板/闸门交互的歧义。

### 1.2 关卡曲线（≥6 关，逐关引入 + 终章合流）

| 关 | 标题 | 引入机制 | 设计意图 |
|---|---|---|---|
| L1 | 起步 First Steps | 纯经典 | 教学：移动、推、撤销、重开 |
| L2 | 薄冰 Thin Ice | 冰面 | 用墙/目标点当刹车，预判滑行 |
| L3 | 深坑 The Gap | 深坑 | 箱子取舍：填坑 vs 达标 |
| L4 | 机关 Mechanism | 压力板 + 闸门 | 用箱子顶门、规划顺序 |
| L5 | 色彩 Hues | 颜色匹配 | 指派 + 路径，避免互相挡道 |
| L6 | 合流 Confluence | 冰 + 坑 + 闸门 + 颜色 | 终章：综合调度 |

### 1.3 反馈与「内涵」

- **撤销 / 重开**：纯逻辑谜题不应有"死局只能重来"的惩罚感。提供无限撤销与一键重开。
- **步数 / 推动数**：记录步数与推动次数，给出每关「最优解参考（par）」（由内置求解器算出）作为可选挑战，而非强制压力。
- **无死亡状态**：玩家不会踩坑致死（坑对玩家=墙），避免突兀重置。失败仅来自「把局面推成无解死局」，可随时撤销。

## 2. 技术栈与架构

> 原则：**少依赖、可静态化、单进程可跑、易部署**（2 核 8G 绰绰有余）。工程上体现"高级"——共享规则引擎、类型安全、服务端可复算校验。

### 2.1 总体形态

单包（single-package）TypeScript 工程，三个逻辑层共享同一套**纯函数规则引擎**：

```
src/engine/   纯 TS 规则引擎（无框架依赖）——前后端共享
  types.ts    数据模型
  level.ts    关卡解析（ASCII + 机关元数据 → 运行态）
  rules.ts    一步移动 / 推动 / 冰滑 / 填坑 / 闸门 / 胜负判定
  levels.ts   6 关数据
  solver.ts   BFS 求解器（验证可解性 + 算 par + 死锁检测）
src/web/      前端（Vite + TS，无 UI 框架，DOM/CSS 渲染）
server/       Fastify 后端（托管静态 + API + 服务端校验 + SQLite 持久化）
scripts/      verify-levels.ts —— 对全部关卡跑求解器的自测脚本
```

### 2.2 前端

- **构建**：Vite + TypeScript，**不引入 React/Vue** —— 谜题渲染是一个小网格，框架是负担。手写轻量渲染层即可，产物更小、首屏更快、也更"克制"。
- **渲染**：**DOM + CSS Grid**，棋子用绝对定位 + `transform` 平滑过渡（`cubic-bezier` 缓动）。相比 Canvas：可访问性好、动画用 CSS 即可、调试简单、像素锐利。
- **美术方向（拒绝 AI 风味）**：遵循极简游戏美学（Monument Valley / Mini Metro：基本几何形、<15 色、留白、外科手术式用色）。
  - **不下载卡通 sprite**，全部用 CSS/SVG 绘制几何形——这恰恰是"高级感"的来源，也保证离线部署、零版权风险。
  - 配色：暖色"纸与墨"基调（米白 `#f4f1ea` 底、暖炭墙、木色箱），机制色克制（冰=冷雾蓝、坑=内凹深色、压力板=内嵌环、彩色箱用低饱和大地色：陶土/鼠尾草/灰蓝/赭石）。
  - 字体：高质量 system serif/grotesque 栈，不依赖 CDN，保证离线与一致性。
  - 交互：键盘（方向键/WASD）、点击/触摸（点目标格自动寻路一步式或方向滑动）、撤销/重开按钮，过渡有"重量感"的缓动。
- **状态**：本地进度（已通关、最佳步数）存 `localStorage`；可选上传到后端排行榜。

### 2.3 后端

- **运行时**：Node.js + **Fastify**（比 Express 更快更现代，依赖少）。单进程同时：
  1. 托管前端构建产物（`dist/`）；
  2. 提供 API：关卡目录、提交解法、最佳成绩排行。
- **服务端权威校验**：后端**复用同一套 `engine`**，对客户端提交的移动序列重放，确认确实通关并算出步数——杜绝伪造成绩，保证"策略完整实现"在服务端也成立。
- **持久化**：**SQLite（better-sqlite3）**——文件型、零外部服务、单机极快，最适合 2 核 8G 小机。存每关最佳成绩 / 排行。
- **构建/部署**：`vite build` → `dist/`（静态）；`esbuild` 把 `server` + `engine` 打成单文件 `dist-server/index.js`；生产仅 `node dist-server/index.js` 一个进程。配 systemd/pm2 + 可选 nginx 反代。

### 2.4 为什么这套"够高级又能过"

- 共享规则引擎 = 客户端体验与服务端校验**同一份真理来源**，逻辑零漂移。
- 内置 BFS 求解器既是**关卡 QA**（保证每关可解、给出 par、检测死锁），也是自测基础设施——直接回应"每关自测、查缺补漏"。
- 全静态前端 + 单 Node 进程 + SQLite = 部署面极小，2 核 8G 轻松。

### 阶段 4–5 决策记录

- **前端零框架**：网格用 CSS Grid，玩家/箱子用绝对定位 + `transform` 过渡——走一步是短过渡，冰面滑行是「按距离拉长的缓出过渡」，沉坑是「滑到坑位 → 缩放淡出 → 移除元素 + 坑标记为已填」。产物仅 ~15.6KB JS（gzip 6.5KB）/ 9.8KB CSS，且**不打包求解器**。
- **服务端权威校验**：`POST /api/scores` 不信任客户端上报的步数，而是用**同一套引擎 `replay()`** 重放提交的移动序列，只有真正通关才入库，步数/推动数以重放结果为准 → 杜绝刷分。
- **修了一个部署暗坑**：原用 `HOST` 环境变量做绑定地址，但很多 shell 会导出 `HOST=主机名`，导致服务端绑定错地址。改名为 `BIND_HOST`，默认 `0.0.0.0`。

### 阶段 2–3 决策记录

- **持久化改用 Node 24 内置 `node:sqlite`**：真 SQLite、零原生编译（免去 `better-sqlite3` 在 Windows 上的构建风险），仅一句实验性告警。
- **求解器从 BFS 升级为 A\***：纯 BFS 在 3 箱以上关卡（L5/L6）状态爆炸打到上限。A\* 用「目标点↔同色箱子的最小指派曼哈顿和」作启发（同时充当死局剪枝：所需箱子全部沉坑 → +∞）。
  - 该启发在**无冰**关上可采纳 → 步数最优；在**冰面**关（L2）会高估（一次推动可滑多格）→ 返回的是「可达的强解」而非理论最短。L2 的真实最优由早期 BFS 证得为 14，已写入 `par`。
- **关卡两处设计 bug 由求解器抓出并修复**：L5 角落死箱（箱子贴墙无法推动）、L6 过于开阔导致搜索爆炸 → 收紧为走廊式终章。

### 4.5 真实浏览器视觉自测（Playwright 截图，端口 8801，已关闭）
用 Playwright（Chromium headless）实跑构建产物并截图菜单 + 全部六关，逐张人眼审查。
- **抓到并修复一个真实视觉 bug**：箱子渲染成「小点」而非木箱。根因——`.crate` 用了 `padding: 11%`，但 `.piece` 是绝对定位元素，**百分比 padding 是相对「包含块（整块棋盘）」而非自身**解析的，于是 padding≈一整格把 body 压没了。玩家方块没事是因为它用绝对定位的 `inset`（相对 piece 解析）。改为给 `.crate .body` 用 `position:absolute; inset:11%`。
- 复测：菜单、L1–L6 均渲染正确——木箱有倒角与重量感、压力板/闸门同组配色、深坑内凹可辨、彩色箱与同色目标环清晰、最宽的 L3（12 格）自适应不溢出。
- 截图脚本通过 `child_process.spawn` 持有 node 真句柄，结束 `child.kill()`；复查 `PORT 8801 CLOSED OK`。Playwright 以 `--no-save` 安装，不进 `package.json`，保持部署依赖精简。

## 5. v2 迭代：折跃门 + 20 关 + 减提示（试玩反馈后）

### 5.1 新机制：折跃门（player-only portal）
- 设计取舍：箱子穿门会引发「推箱过门」的歧义（玩家进门口会被传走、与刚传出的箱子撞位）。为保持**确定性**，定为**只传送玩家、箱子把折跃门当墙**。这反而是更纯的探索机制——「我够不到这个箱子的可推一侧，得先借门绕过去」。
- 引擎改动：`Cell.portal` + `Level.portalPartner[]`；`crateCanEnter` 对折跃门返回 false；玩家踏入折跃门→落到同色伙伴格（伙伴格须可入）。渲染用同心环 + 玩家瞬移的 `warp` 动画（不跨屏滑行）。求解器经 `applyMove` 自动兼容。

### 5.2 关卡：6 → 20（六章），靠求解器迭代
- 用一个临时单图测试器（`tsx` + 求解器，<1s/图）反复试错，再把通过的图并入目录。**首轮 20 张草稿 14 张不可解**——常见坑：贴墙角的死箱、顶行的箱子无法下推、彩箱在角、目标数≠箱数、单宽走廊里箱子堵死折跃门路。逐一修复至全绿。
- 较难的几章锚定在 v1 已验证的可解骨架上（坑：moat；闸门：顶门/双枢；色彩：开阔互换），再加障碍/不规则外形与组合，控制状态空间在求解器可达范围内（≤~54 万状态）。
- 难度策略：**靠紧凑+死局+非显然顺序**做人类难度，而非拉长步数（保证求解器可判定）。par 跨度 10～45。

### 5.3 减提示 + 章节菜单 + 图例
